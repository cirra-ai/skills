# Security Benchmark for Salesforce (SBS) Audit

Run a read-only audit of a Salesforce org against the
[Security Benchmark for Salesforce (SBS)](https://docs.securitybenchmark.org/) —
a vendor-neutral compliance standard for Salesforce security, licensed
CC BY-SA 4.0. This skill wraps the Cirra AI MCP Server's `sbs_audit` tool.

**When to use this skill** — only when the user explicitly asks for an SBS
audit ("audit against SBS", "is this org SBS-compliant?", "check SBS-AUTH-001",
"sweep all Critical SBS controls"). **Do not** invoke as a side effect of
routine metadata work, prototyping, or legacy artifact management. SBS auditing
is opt-in by design.

**What this skill is not** — it is not an enforcement layer. The audit never
blocks, modifies, or mutates anything. It reports findings; remediation is the
user's call.

## Dispatch

Parse `$ARGUMENTS` to determine the audit scope:

| First argument or intent                               | Action                                           |
| ------------------------------------------------------ | ------------------------------------------------ |
| `control SBS-XXX-NNN` or a bare `SBS-XXX-NNN` ID       | Call `sbs_audit` with `control_ids: ["<id>"]`    |
| `category <code>` or one of `ACS`/`AUTH`/.../`SECCONF` | Call `sbs_audit` with `categories: ["<code>"]`   |
| `risk Critical`/`High`/`Moderate`                      | Call `sbs_audit` with `risk_levels: ["<level>"]` |
| `all` or no argument                                   | Call `sbs_audit` with no filters (every control) |

Filters compose: e.g. `category OAUTH risk Critical` → `{categories: ["OAUTH"], risk_levels: ["Critical"]}`.

When the intent is missing or unclear, **use `AskUserQuestion`** before
running:

```
AskUserQuestion(question="What scope should I audit?\n\n1. **One control** — SBS-XXX-NNN\n2. **A category** — Access Controls, Authentication, OAuth, etc.\n3. **A risk band** — Critical / High / Moderate only\n4. **Everything** — sweep every control in the benchmark")
```

A full sweep is fine, but until check coverage grows, many or all selected
controls will come back `not_implemented`. The response's
`coverage.checks_implemented` vs `coverage.controls_total` give the live
ratio — surface them so the user understands what `not_implemented` means in
context.

## Result handling

`sbs_audit` returns:

- `source` — attribution block (SBS, CC BY-SA 4.0, upstream commit). Always
  include this in your summary to the user.
- `coverage.controls_total` / `coverage.checks_implemented` — how many of the
  total controls our tool actually checks today. Surface this when responding
  to broad sweeps so the user knows what `not_implemented` means.
- `counts` — `{ selected, pass, fail, not_implemented }`.
- `results[]` — per-control entries with `status`, `risk_level`, `category`,
  `remediation.scope`, `doc_url`, and (for `fail`) a `findings` array.

### Reporting

1. Start with a one-line headline: how many `fail` / `pass` / `not_implemented`
   out of `selected`, plus a reminder that `not_implemented` means "we don't
   check this yet" — not "this is fine".
2. List **`fail` first**, sorted Critical → High → Moderate, with the
   `doc_url` so the user can read the upstream audit procedure and remediation
   for each finding.
3. Group `not_implemented` by category in a single collapsed/short list rather
   than reproducing them in detail.
4. Always close with the attribution: e.g.
   _Based on Security Benchmark for Salesforce (SBS,
   https://docs.securitybenchmark.org/), CC BY-SA 4.0, upstream commit `<sha>`._

### What NOT to do

- **Do not paraphrase, summarize, or quote SBS control descriptions, audit
  procedures, or remediation language.** The benchmark prose is CC BY-SA 4.0
  and lives upstream. Link to `doc_url` for the details.
- **Do not invent control IDs.** If the user names a control we don't have
  (e.g. `SBS-XYZ-999`), call `sbs_audit` anyway with `control_ids: [...]` and
  let the response show it as missing rather than fabricating a finding.
- **Do not edit the vendored data files** in the cloud-app's `src/data/sbs/`.
  They are CC BY-SA 4.0 and are regenerated by `scripts/fetchSbs.js`.
- **Do not enforce or block other operations** based on findings unless the
  user explicitly asks ("Layer 3" enforcement is a separate, opt-in feature
  not part of this skill).

## Tool reference

- `sbs_audit` — read-only audit handler. Parameters: `control_ids?: string[]`,
  `categories?: string[]`, `risk_levels?: string[]`, `sf_user?: string`. See
  the MCP tool's `description` and `instructions` for the canonical contract.
